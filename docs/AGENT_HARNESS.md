# Agent Harness

How unattended, isolated Claude Code sessions pick up work on this repo — from a GitHub issue to a merged PR — without a human watching in real time.

## Why this exists

This repo's own commit history is five rounds of bugs that only ever showed up when actually run against live Amazon and live SMTP — a wrong CSS selector, a DOM virtualization race, a flaky first-party API, a DNS library picking a random address, a port/TLS mismatch. None of these were visible from reading the code. That history motivates the two constraints this harness is built around:

1. **Don't burn live Amazon requests carelessly.** Repeated rapid runs during this project's initial debugging appeared to trigger Amazon rate-limiting or throttling — items that had valid prices in one run came back with no price data moments later. A pipeline that iterates by re-running against live Amazon on every change is both slow and actively risks degrading the thing it's testing.
2. **Don't let an unattended agent land something broken with nobody watching.** The whole point of this harness is that no human is in the loop turn-by-turn — so the checks that would normally happen via conversation need to happen structurally instead.

**The one principle everything below follows from: scrutiny scales with risk, not with pipeline position.** A pure-logic change (e.g. the notification-dedup work in issue #1 — pure JSON/date-math, no network) should move through this pipeline fast and cheap: cheap models, no live checks, minimal review overhead. A change touching the scraping/DOM/SMTP code called out in `CLAUDE.md` should get independent plan review, a live spot-check, and a stronger model — every time, without exception. This shows up in three separate places below (which stages get invoked, which model runs them, whether a live Amazon call is authorized) — it's one rule applied three times, not three different rules.

## Where brainstorming ends and this pipeline begins

**Brainstorming — turning a rough idea into a well-specified issue — is not part of this pipeline. It's synchronous, interactive work that happens before the pipeline is ever triggered**, the same way issues #1 through #3 in this repo were written: a human working with Claude Code in conversation, refining a rough idea through questions until it's specific enough to hand off. Everything from here on — Analyzer through the final PR Risk Analyzer script run — assumes that work is already done and runs unattended, via GitHub Actions, starting from an issue number. The Analyzer's job is to *confirm* the issue is complete, not to *make* it complete; if it isn't, the pipeline stops and asks, it doesn't try to brainstorm on its own.

## Where this builds on [Superpowers](https://github.com/obra/superpowers) instead of reinventing it

[obra/superpowers](https://github.com/obra/superpowers) is a cross-platform, installable Claude Code skills plugin implementing a similar-in-spirit chain — brainstorming → git worktrees → planning → subagent-per-task development with two-stage review → TDD → code review → branch finishing. Two independent designs landing on the same shape is a good sign the shape is right, and there's no reason to hand-roll what it already does well:

- **Planning** should use Superpowers' `writing-plans` skill for the actual task-breakdown mechanics (bite-sized tasks, exact file paths, verification steps), rather than us maintaining a competing format. Our own addition on top — the coverage checklist mapping every plan item back to the source issue's acceptance criteria — stays ours, because it's specific to this pipeline being issue-driven, which a general planning skill has no way to know about. One override: the skill's own default save location (`docs/superpowers/plans/YYYY-MM-DD-<name>.md`) is ignored — this pipeline's plan always lands at `.agents/plan.md`, since every other stage and the Deployer's pre-squash cleanup depend on that exact path.
- **TDD execution** should invoke Superpowers' `test-driven-development` skill directly for RED-GREEN-REFACTOR, rather than `implementer.md` re-explaining the same philosophy in different words.
- **Review** should invoke Superpowers' `requesting-code-review` / `receiving-code-review` skills for the two-stage per-task review, rather than us defining a competing severity rubric.
- **Implementation is a fresh Implementer job per task**, matching the *pattern* Superpowers' `subagent-driven-development` skill validates — fresh subagent per task, two-stage review — with a two-stage review (spec compliance, then code quality) after each task before the next one starts — see "The Implementer," below. **We do not use that skill's own dispatch mechanism** (it's a bash-script-driven controller pattern — task briefs and review packages as files, invoked via shell commands — built for a human or script shelling out locally). Our dispatch is a GitHub Actions `strategy.matrix` job, one job per task, with git commits as the handoff (see "Git as the inter-stage handoff mechanism," below) — mechanically different, same shape, and the one that actually fits running unattended inside GitHub Actions.
- **Branch finishing gets a real choice**, not an assumed default: merge (not available yet — see the auto-merge non-goal), open a PR, keep the branch without a PR, or discard — using Superpowers' `finishing-a-development-branch` skill for that decision tree and the cleanup mechanics, rather than us reimplementing it. See "Deployer," below.
- **Any bug fix — the Implementer's one bounded fix attempt against a review finding, and the Tester's one fix attempt for a real bug — uses Superpowers' `systematic-debugging` skill** to root-cause before patching, and **`verification-before-completion`** before either stage declares things done. Both stages already had a "one bounded attempt, no speculative back-and-forth" rule; these skills are how that one attempt is actually executed well, not a new rule layered on top of it.

**What stays ours, deliberately, because it's either specific to this project or Superpowers' README doesn't cover it at all:** the risk-scaled *conditional* invocation of stages (Superpowers' gates read as uniform, always-on; ours skip stages entirely for low-stakes work), model and cost tuning per stage (no mention of this in Superpowers), the single-live-external-API-hit-per-run constraint (specific to this project's actual Amazon-throttling history), the GitHub-Actions-native unattended trigger/resume mechanics below (Superpowers reads as built for a human answering questions in real time, not a red/green Actions run resumed via issue comments hours later), and the PR Risk Analyzer's shadow-mode auto-merge track record.

**No separate whole-diff code-review stage was added, on purpose.** It might look like a gap next to Superpowers' dedicated code-review step, but the two-stage per-task review (below) already covers code quality at the point where it's cheapest to fix, Plan Validator already covers plan-level soundness before any code exists, and PR Risk Analyzer already does a final holistic pass for merge-risk purposes. A fourth review layer on top of those three would be redundant, not additive.

### Installing Superpowers (verified against the real repo, not assumed)

`obra/superpowers` is itself a self-contained Claude Code plugin — its repo root carries `.claude-plugin/plugin.json` directly (currently pinned at tag `v6.1.1`), not just a skills folder needing a wrapper. That means two different install paths make sense for the two different ways this repo runs Claude Code, and neither is a guess:

- **Interactive/human sessions** (someone with this repo open locally): this repo's `.claude/settings.json` declares Superpowers' marketplace under `extraKnownMarketplaces` (the documented "team marketplaces" mechanism — see Claude Code's plugin docs). Opening this repo in a trusted Claude Code session prompts to install it from there; a human confirms once.
- **The CI pipeline** (`agent-pipeline.yml`): marketplace-based install depends on an interactive trust prompt that doesn't exist in an unattended job. The Claude Code Action turns out to have first-class inputs for exactly this, so any job's `claude-code-action` step that needs a Superpowers skill uses those directly rather than a CLI flag threaded through `claude_args`: `plugin_marketplaces: https://github.com/obra/superpowers-marketplace.git` and `plugins: superpowers@superpowers-marketplace`. No marketplace registration step, no trust prompt — the action adds the marketplace and installs the plugin before running Claude, every time, deterministically.

## Risk classification (the thing everything else branches on)

Before any other stage runs, the Analyzer classifies the task as one of:

- **Pure-logic** — touches only application state/config/logic with no DOM parsing or SMTP changes (e.g. the dedup/pruning feature in issue #1).
- **Scraping-touching** — touches `scrapeWishlist()`, its selectors, the scroll/merge/retry logic, or anything in `sendEmail()`'s connection handling.

This classification gates: whether the Plan Validator runs at all, whether the Tester is authorized to spend its one live Amazon check, and whether the PR Risk Analyzer's rubric can even consider the change low-risk (scraping-touching changes are never low-risk, regardless of test coverage — see the rubric below).

## How this actually runs

**One workflow file, `.github/workflows/agent-pipeline.yml`, triggered exclusively via the Actions tab's "Run workflow" button — `workflow_dispatch` with a single input, `issue_number`.** No label triggers, no `issue_comment` triggers, no second workflow watching for anything. Both starting a fresh pipeline run and resuming one that stopped go through the exact same form, with the exact same one field.

- `run-name: "Pipeline: issue #${{ inputs.issue_number }}"` so parallel runs for different issues are distinguishable at a glance in the Actions list, rather than a generic name you have to click into to identify.
- `concurrency: { group: pipeline-${{ inputs.issue_number }} }` so accidentally clicking Run twice on the same issue queues the second attempt instead of two runs racing on the same branch.

**Every pipeline stage is its own GitHub Actions job, running its own top-level `claude-code-action` session directly against `.claude/agents/*.md`** — Analyzer, Planner, Plan Validator (conditional), Implementer (fanned out via a `strategy.matrix` populated from the Planner's own task list — `fromJson(needs.planner.outputs.tasks)`, `max-parallel: 1` to preserve the sequential per-task dependency, `fail-fast: true`), Tester, Deployer, and — on the live-check path — a second `live-amazon-check` job followed by its own `deployer-after-live-check` job. There is no long-running orchestrating session and no in-process subagent dispatch: each job checks out the same feature branch (already carrying every earlier stage's commits) and starts completely fresh.

**This replaced an earlier single-session design** — one job, one Claude Code session acting as an "Orchestrator" that dispatched every stage in-process via the `Agent` tool, one call per stage. Two independently-triggerable-workflow-files and multi-job-DAG-with-shared-session designs were tried before that one too; all three gave up real usability (parallel-run visibility, a single obvious "restart" action, clean per-stage cost attribution) for boundary guarantees this project's actual risk profile didn't need as much as it needed those things. What actually forced the move to the current design was cost and time, not tooling taste — measured directly, not assumed:

- **The single-session Orchestrator's own ever-growing context was the dominant cost driver, not the actual per-stage work.** Every orchestrator-level turn re-read its entire accumulated history via prompt cache, so cost compounded with how far into the run a turn happened, not with how much real work that turn's stage did. Reconciled exactly against real billed cost from issue #14's dogfooded single-session run: of the Sonnet portion of that run's $3.21 total, **cache writes and cache reads together accounted for ~74% of the cost** — and cache *writes* (1.25× base rate, triggered by the growing conversation repeatedly shifting the cache breakpoint) cost more than cache *reads* (0.1× base rate), the opposite of what you'd expect from "caching mostly saves money."
- **Splitting stages into separate jobs was prototyped first on a 2-job slice** (Analyzer + Planner only, against issue #17) before committing to the full 8-job version, specifically to confirm the theory with real measured data rather than assumption. Result: Planner took 124s split vs. 327.8s (5m28s) for the identical stage inside the single-session pipeline — less than half the time — because the single-session Planner was the *second* stage in an already-growing conversation and paid to re-read all of that history before doing any real work; the split version started fresh with nothing to re-read. The job-graph UX also worked as hoped: named nodes, each with its own real status/duration/log, no digging through one opaque blob required.
- **Per-stage cost attribution went from an estimate to a fact.** In the single-session design, a per-stage token/tool-use/duration table (captured from the `Agent` tool's own `<usage>` block) was the best available signal, and it was explicitly labeled a *relative resource-usage* signal, not a dollar split — a genuinely accurate per-stage dollar figure wasn't reconstructable from inside one compounding session. In the split design, each stage's `total_cost_usd` is directly knowable the moment its one `claude-code-action` step ends, because that step *is* the whole job. See "Observability," below.

The one place a hard, GitHub-enforced boundary still matters — the live Amazon check — gets one, described below. Splitting every stage into its own job as a side effect also means every stage now gets its own real, GitHub-enforced permission scope (a `permissions:` block per job) — something the old single-session design structurally couldn't offer, since job/step permission scoping needs a job boundary GitHub Actions can enforce, and there mostly wasn't one there.

### Fresh start vs. resume — the facts are precomputed, the judgment stays per-job

Every run starts identically: you fill in `issue_number` and click Run. What happens next depends on repo state, not on anything you chose in the form:

1. Resolve the branch name from the issue number (`issue-<N>-...`).
2. **Branch doesn't exist** → fresh start. The Analyzer job creates it from `main`, begins there.
3. **Branch exists** → resume. The Analyzer job checks out the branch and reads two things:
   - Which `.agents/*.md` files are already committed, which tells it which stages already finished.
   - The issue's comment thread: find the most recent comment *this pipeline* posted (always carrying a recognizable `🤖 **Pipeline checkpoint` marker so it can find its own last checkpoint, and always posted as `claude[bot]`), then read anything posted after that as this run's guidance.
4. If there's no new comment since the last checkpoint, there's nothing to fold in — Analyzer just retries whatever stage stopped, plain, no correction. If you did reply, that reply is the correction for this run.
5. **A resumed branch also gets synced with `origin/main` before any other job checks it out.** A resumed branch can be arbitrarily old, and pipeline-infra scripts added to `main` since it was created (`scripts/*.js`) would otherwise simply be missing from its tree — see the debugging-history section below for the real `MODULE_NOT_FOUND` this caused. Squash-merge-only (see below) means this merge commit never surfaces in the final PR either way. A fresh-start branch needs no such sync, since it's created from current `main` moments later.

**All of the above are mechanical facts, not judgment calls, so a script computes them before the Analyzer job's own Claude session even starts** — `scripts/resolve-pipeline-state.js`, run as a plain step ahead of the `claude-code-action` invocation. It slugs the branch name from the issue title, checks whether a branch matching `issue-<N>-*` already exists (matching by prefix, since an earlier run's chosen slug can differ slightly from what a fresh slugging pass computes — the branch with real commits on it is the one that matters, not exact string equality), reads the `.agents/*.md` file list straight from that branch's real tree via the GitHub API, and finds any issue comments postdating the last `claude[bot]` checkpoint comment. These come in as precomputed facts in the Analyzer's prompt (`BRANCH`, `THIS RUN IS A: FRESH START/RESUME`, the `.agents` file list, the new-comment text) instead of being re-derived via tool calls every single run.

**What's still a judgment call, deliberately not automated away:** whether to trust the precomputed `.agents` file list over something a checkpoint comment claims (the script's answer already wins by construction, since it reads the real tree rather than prose — but the Analyzer is told explicitly to say so if it finds a real discrepancy once it looks at `git log` itself, rather than silently trusting either source), how to actually fold new guidance into what it does next, and what to do about an ambiguous case like multiple branches matching the same issue number (the script logs a warning to the job's own output for this rather than silently picking one and hiding the ambiguity). The mechanical parts are the branch name, its existence, and what's literally on disk; deciding what any of that *means* for this run stays a judgment call.

**Known, deliberate gap: no skip-if-already-done resume logic.** Every stage job uses `scripts/resolve-pipeline-state.js` for the fresh-start-vs-resume branch facts, but nothing looks at "which `.agents/*.md` files already exist" and skips stages that already finished — a re-run after a mid-pipeline failure currently redoes every stage from Analyzer onward, not just the one that stopped. In practice, later stages tend to notice and self-correct rather than blindly redo work — see the "Planner reports zero tasks" case in the debugging-history section below — but that's a stage being sensible about stale state, not a designed resume mechanism. Solving this properly (each job checking whether its own artifact already exists, and if so reading committed state instead of re-dispatching) is a separate, meaningfully-sized problem — see "Open questions," below.

**This is the entire input mechanism.** There's no separate "guidance" field on the dispatch form — guidance lives in the issue thread. The Actions button is the "go" signal; the comment thread is the "how" signal. When something stops the pipeline (a stage needs clarification, a downstream stage rejects an upstream one, the live-check approval is pending), the relevant job posts a comment on the issue (or the PR, once Deployer has opened one) explaining what happened and what it needs — then the job ends. You reply on that thread, go back to the Actions tab, run the same workflow with the same issue number. Repeat until it completes.

**Whether the run is green or red is the signal for whether it needs you** — with one platform-level caveat worth knowing about: a `strategy.matrix` job whose array evaluates to zero entries (the Implementer job, when the Planner legitimately reports zero remaining tasks) marks the *entire workflow run* as "failure" in GitHub's own top-level status, even when every job that actually ran succeeded and the zero-entry case was itself correct and intentional. This is guarded against structurally now (`implementer` has an explicit `if:` so an empty task list skips the whole job the ordinary way, rather than asking GitHub Actions to expand a zero-entry matrix at all) — but if you ever see a red run-level badge with an all-green job list underneath it, check for this before assuming something actually broke. Absent that specific case, full completion (PR opened, or an update pushed to an existing one) is success, and stopping early for any other reason — needs clarification, a stage rejected another stage's work, waiting on the live-check approval below — is a genuinely failed run. Red in the Actions tab (that specific quirk aside) means "go read the issue," never "something is silently fine."

### The one hard boundary: live Amazon requires human approval, enforced by GitHub

Every stage already runs in its own job with its own scoped permissions (see above) — but tool-list restriction and job-permission scoping both reduce the *surface* for a stage to reach live Amazon or attempt a merge; a Bash-equipped agent can technically still run arbitrary commands within its own job. Tool omission is not a hard sandbox. For live Amazon specifically, this pipeline doesn't rely on that surface reduction alone — it gets a real, separate mechanism instead.

- **If the Tester decides no live check is warranted** — the common case, since most work on this repo is pure-logic — the pipeline just continues through Deployer and then the PR Risk Analyzer script, to a real PR. There's no hard boundary on that path, so there's no reason to stop and wait for a human to manually continue something that doesn't need approval.
- **If the Tester decides a live check IS genuinely warranted** — per the risk classification and the fixture corpus being insufficient — it says so (a job output) and the pipeline stops right there on that path, without touching live Amazon itself.
- A second job, gated with `environment: live-amazon-check` (a GitHub Environment configured with required reviewers), only runs `if:` that output says a live check is needed. GitHub itself pauses this job and shows a "Review deployments" approval prompt — a real, GitHub-enforced checkpoint, not a convention — before anything in it executes.
- Once approved, that job triggers `check-wishlist.yml` via `workflow_dispatch`, waits for it, and pulls the resulting `wishlist-debug.html` into `test/fixtures/` as a new fixture regardless of pass/fail — then a following `deployer-after-live-check` job runs the remaining stages (Deployer, then the PR Risk Analyzer script + cost report) itself. It can do this cleanly because it checks out the same branch, which already has everything the earlier stages committed.
- This still respects "at most once per pipeline run": the Tester only sets that output once, and only when it's actually warranted, not as a default.

### A caveat worth being honest about

The three genuinely hard guarantees in this design are the environment-gated approval above (for live Amazon), GitHub branch protection requiring human review before merge (for Deployer, below), and GitHub's own refusal to let this pipeline's credentials touch `.github/workflows/*` (below). Everything else here is a strong convention — per-job permission scoping, tool-list restriction — plus PR-level review, not a technical impossibility. Don't oversell this to yourself when extending the harness — say so explicitly if a future addition is a convention rather than an enforced boundary.

This is also why every `claude-code-action` invocation across this pipeline runs with `--permission-mode bypassPermissions`: the alternative, Claude Code's normal interactive permission-approval prompt, has no human to answer it in an unattended Actions job — the first real run against issue #1 (under the earlier single-session design) confirmed this the hard way, dead-ending on "This command requires approval" for `gh`, `WebFetch`, `git ls-remote`, and even dispatching a stage, without ever reaching a stage that does real work. Bypassing that prompt at the top level of each job is what makes unattended execution possible at all; it does not bypass each job's own `tools:` frontmatter restriction (a fresh Tester dispatch still literally isn't given a live-network tool, regardless of its own permission mode) — that restriction and the three hard guarantees above remain the actual backstops.

### The third hard boundary: this pipeline cannot touch `.github/workflows/*`, by design

A real dogfooded run (under the earlier single-session design, against issue #1) hit this directly: a task in its plan needed a small, genuinely-necessary change to `check-wishlist.yml` (a step to commit `notified.json` after a run that changed it — the same "commit a file back to git instead of a database" idiom this whole harness already runs on). The push was rejected — not by anything in this repo's own `permissions:` blocks, but by GitHub itself: `refusing to allow a GitHub App to create or update workflow .github/workflows/check-wishlist.yml without workflows permission`. The Claude GitHub App installation this pipeline runs under doesn't have that scope, and this is deliberate, not a bug to route around by granting it:

- **Workflow files are the highest-blast-radius thing an unattended agent in this repo could touch** — they run with secrets access and can rewrite their own guardrails, including this very pipeline's `permissions:` blocks and the live-Amazon-check approval gate. An agent that can edit `.github/workflows/*` unattended can, in principle, quietly edit away every other boundary this doc describes.
- **The pipeline correctly identified this as a wall, not a bug**, and stopped to ask rather than working around it — exactly the "red means needs-human" behavior the rest of this design is built on. This is the system working, not malfunctioning.
- **The fix is a human (or an interactive Claude Code session, credentials permitting) applying the specific diff by hand** — usually already fully spelled out in `.agents/plan.md`, since the Planner writes it whether or not the Implementer can land it — and pushing it to the branch. **Title the commit `Manual (#<issue-number>): (<task-number>/<total-tasks>) <summary>.`** — same shape every stage's commits use, but `Manual` instead of an agent's name, since attributing this one to the Implementer would misrepresent who actually made it. This both gives a resumed run a direct signal that the task is done, rather than having to infer it by diffing the file against the plan, and keeps the issue page's commit trail honest about which changes were agent-made and which weren't.
- Then re-run `agent-pipeline.yml` with the same issue number — same resume mechanics as any other stop, no special case needed.

## Pipeline stages

| Stage | Subagent file | Default model | Live Amazon? |
|---|---|---|---|
| Analyzer | `.claude/agents/analyzer.md` | Haiku 4.5, `effort: low` | No |
| Planner | `.claude/agents/planner.md` | Sonnet 5 | No |
| Plan Validator | `.claude/agents/plan-validator.md` | Sonnet 5 | No — only invoked when the Analyzer classified the task scraping-touching |
| Implementer | `.claude/agents/implementer.md` | Sonnet 5 | No — a separate job per task in the plan, not one job for the whole feature |
| Tester | `.claude/agents/tester.md` | Sonnet 5 | Decides *whether* a live check is warranted; does not perform it itself — see the approval-gated job above |
| Deployer | `.claude/agents/deployer.md` | Sonnet 5 | No — opens/updates the PR, does not merge |
| PR Risk Analyzer | `scripts/pr-risk-check.js` — not a subagent, no model call at all | N/A | No |

Haiku is deliberately used only for the one stage that's genuinely *mechanical* rather than open-ended judgment: confirming a well-specified issue is complete. PR Risk Analyzer used to be Haiku's other mechanical-rubric stage, but "mechanical" turned out to mean "actually deterministic" once looked at closely — see its own section below for why that became a plain script instead of a model call at any tier. Planner, Implementer, and Tester never downgrade below Sonnet 5: a cheap model on the Planner in particular tends to cost more overall, since a bad plan poisons every stage that reads it.

**Model choice is set per job in `agent-pipeline.yml`'s `claude_args`, not in the subagent file's own frontmatter default** — each stage's job hardcodes the model it needs (`--model claude-haiku-4-5` for Analyzer, `--model claude-sonnet-5` for everything else), which is the split-architecture equivalent of the single-session design's old "override at the dispatch call site" mechanism.

## Failure handling

Different failure modes get different responses — treating them all the same either under-reacts (silently plowing through a real problem) or over-reacts (building retry infrastructure for things that already have a retry mechanism underneath them).

1. **Infra-level crash** (API error, runner blip, timeout) — no custom logic. The Anthropic SDK already retries transient errors automatically, and above that, re-running the same workflow with the same issue number (see "Fresh start vs. resume" above) picks up via git — though see the resume-logic gap noted above: this currently means redoing every stage from Analyzer onward, not resuming mid-pipeline cleanly.
2. **A stage can self-correct within its own scope** (the Tester finding and fixing an obvious bug is one example; a per-task Implementer job addressing its own two-stage review findings is another) — bounded to **one attempt**, then stop. Scoping this to a single task rather than the whole feature is one of the benefits of the fresh-job-per-task model — a stage that keeps trying increasingly speculative fixes across many iterations is a real cost and correctness risk, and a smaller unit of work makes "did the one fix actually hold" a much easier question to answer cleanly. If the fix doesn't hold, that's a signal to surface, not to keep guessing at.
3. **A downstream stage rejects an upstream stage's work in a way it can't fix itself** (Plan Validator rejects the plan outright; Tester finds something fundamentally broken, not a one-line fix) — **does not auto-loop back to re-run the earlier stage.** This is deliberate: a downstream stage disagreeing with an upstream one is itself a risk signal, and this harness is built around escalating risk signals to a human rather than having agents resolve disagreements among themselves. The job stops with a comment explaining the disagreement; a human decides how to proceed (re-run with guidance, fix the branch by hand, or restart from an earlier stage).
4. **The live-check job itself fails** (the triggered `check-wishlist.yml` run errors out, or comes back with a result that doesn't resolve the question the Tester needed answered) — this is a terminal result to report, not a reason to trigger the live check again. "At most once" means once even when that one attempt comes back bad.
5. **A stage needs human clarification** (the Analyzer's core job) — stops rather than guessing, with the specific gap in both its `.agents/analysis.md` commit and the issue comment.
6. **A stage hits a real platform restriction it has no way to cross** (the concrete case so far: GitHub refusing any push touching `.github/workflows/*` from this pipeline's credentials — see "The third hard boundary," above) — not a bug to route around, and not something to keep retrying. Stop, name the exact blocker and the exact diff needed in the issue comment, and wait for a human to apply it (with an explicit "applied manually" commit message) and re-trigger.

## Snapshot/fixture testing (how iteration avoids hitting live Amazon)

The workflow's `SAVE_DEBUG_ARTIFACTS=true` step already produces exactly what's needed here: a full DOM dump (`wishlist-debug.html`) of a real Amazon wishlist page. Puppeteer's `page.setContent(html)` loads a static HTML string into a page with zero network activity, and the same extraction code (`extractCurrentItems`, the selectors, etc.) runs against it identically to how it runs against a live page.

- **Fixture corpus** lives at `test/fixtures/*.html`. Each fixture is a real `wishlist-debug.html` captured from an actual run, committed with a name describing the scenario it covers (e.g. `partial-price-failure.html` for the "10 items missing price" scenario that motivated the reload-retry logic).
- **What fixtures can and can't test.** A static snapshot validates parsing/selector logic against a frozen DOM state. It cannot reproduce Amazon's *live* virtualization behavior (a dynamic client-side race, not a DOM state) — that logic is validated by the fact that it exists for a documented, live-diagnosed reason (see `CLAUDE.md`), not by a fixture pretending to simulate it.
- **Capture-once-replay-forever.** Any time the approval-gated live check produces a new `wishlist-debug.html`, it's saved as a new fixture regardless of whether the run passed or failed. The corpus should only ever grow; future pipelines need live Amazon less often as a result.
- **Pure-logic changes need zero fixtures and zero live calls.** The dedup/pruning feature in issue #1 is a good example — it's pure JSON + date-math operating on an already-scraped array, fully unit-testable with `node:test` and no browser at all.

## Git as the inter-stage handoff mechanism

Every stage-job commits its own work to the shared feature branch, with a substantive commit message — not just code changes, but a record of *why* — **and pushes it immediately, every time, not just at the end of the run.** This is not optional polish: the pipeline runs on ephemeral GitHub Actions runners, and a commit that only exists in a runner's local working copy is gone the instant that job ends. A real dogfooded run under the earlier single-session design lost a full pipeline pass this way — several stages genuinely ran, tests genuinely passed, and none of it was ever pushed, because the only place that design originally said "push" was the Deployer, which never runs in the (common) case where no live check is warranted. Every stage now pushes its own commits; don't reintroduce a single point of failure here by consolidating this back down to one stage's job.

**Every commit any stage makes is titled `{Stage name} (#<issue-number>): <summary>.`** — e.g. `Analyzer (#1): Scope confirmation and risk classification.`, `Implementer (#1): (3/8) Add isSuppressed/partitionMatches for the 14-day suppression window.` (the Implementer additionally carries its task position and the plan's total task count, both already read from `.agents/plan.md`). The point isn't cosmetic: GitHub cross-links any `#N` onto that issue's own page, so a consistent, actor-prefixed, ordered title turns the issue page into a readable trace of who did what, in what order — not just the branch's commit history. Each job's prompt states the issue number and the exact commit-title format explicitly, and each subagent file's own commit instruction states its exact title format too, so it doesn't depend on anything being remembered mid-run. A step that isn't a subagent at all (the live-check job's own fixture commit, described below) uses the same shape with `Live Check` as its actor name; a workflow-file diff a human had to apply by hand (see "The third hard boundary," above) uses `Manual` — the convention covers every commit on the branch, not just the ones an agent made.

- Analyzer commits `.agents/analysis.md` (scope confirmation, risk classification, any caveats found).
- Planner commits `.agents/plan.md` (the seams it's introducing, plus the task breakdown and coverage checklist — see "The Planner" below).
- Plan Validator (when invoked) commits `.agents/plan-validation.md`.
- Each per-task Implementer job commits its own code + tests as separate red/green/refactor commits — the commit history *is* the TDD record, which is the reason to do this in git rather than a scratch file. Across a multi-task plan, the branch ends up with one red/green/refactor cluster per task, in order, which is itself a readable trace of how the feature was actually built.
- Tester commits any fixes it had to make.
- Deployer commits the pre-squash `.agents/` cleanup, then opens/updates the PR.
- PR Risk Analyzer (`scripts/pr-risk-check.js`, run after Deployer since it needs a real PR number) commits its append to `.github/pr-risk-log.jsonl` itself and pushes directly, as a plain workflow step in the Deployer job — not a subagent, no `claude-code-action` step of its own.

**Why git and not a shared context or a state file:** this is the same "no database, commit files" idiom the project already uses for `notified.json` (see issue #1), applied one level up to pipeline coordination instead of app runtime state. It gives a fresh job — one with zero memory of any earlier stage, picking up seconds or hours later, resuming automatically inside the approval-gated live-check job — a durable, inspectable record: `git log --stat` on the branch shows exactly which stages have completed and what each one found, with no separate state file that could drift out of sync with reality.

**Squash-merge only, no other technique.** A GitHub squash merge captures the branch's *final* tree state as one diff against `main` — it discards all intermediate history in the process. So:

- The Deployer's last action before requesting merge is `git rm -r .agents/` and a commit — this must happen *before* the squash, since squash only cares about the tree state at merge time. Get the ordering wrong and `.agents/` permanently lands in `main`.
- No history rewriting on the branch, ever (no `git commit --amend`, no interactive rebase) — matches this environment's existing git-safety norms. If a later stage finds an earlier stage's work wrong, that's a new commit, not an edit to history.
- The full stage-by-stage trail survives on the branch for as long as the branch exists (visible in the PR's Commits tab), but that's best-effort, not permanent — once the branch is deleted post-merge, those commits are no longer reachable from any ref and could eventually be garbage collected. The one thing guaranteed to survive forever is the **squash commit's message on `main`**, which is why the Deployer's actual job is composing a good PR description (see below) rather than relying on branch history sticking around.

## The Planner (TDD, specifically)

The Planner's job is not "write tests then implement" — that's too vague for a downstream agent with no back-and-forth to execute deterministically. It uses Superpowers' plan-writing skill for the actual task breakdown — bite-sized tasks (a few minutes of work each), exact file paths, verification steps — rather than us maintaining a competing format for something a general-purpose skill already does well. On top of that mechanic, its deliverable `.agents/plan.md` has two things that are specific to this pipeline being issue-driven, which a general skill has no way to know to include:

1. **Seams.** Which pure functions get extracted, with signatures, before any test is written — named explicitly enough that each task in the breakdown can be handed to a fresh Implementer job with no further context from the Planner.
2. **Coverage checklist.** Every acceptance criterion in the source issue mapped to at least one task/test in the breakdown above, plus an explicit list of what the plan does *not* touch, checked against the issue's stated non-goals. This is the Planner's own self-check, required on every plan regardless of risk level (see Plan Validator below for when a second, independent check also happens).

The Planner also reports its task breakdown as structured output (`tasks`: a JSON array of `{index, total, description}` objects) — this is what the Implementer job's `strategy.matrix` fans out over downstream, so it's read mechanically, not re-derived by a human.

## Plan Validator

Only invoked when the Analyzer classified the task as scraping-touching. The case for a *separate* job here rather than folding this into the Planner's own self-check is objectivity: the same reasoning that produced a flawed plan is often blind to that exact flaw on a re-read, which is why code review is a different person rather than the same author reading their own diff again. Running this on every trivial change would be flat overhead with little payoff, since most of what it checks (does every acceptance criterion map to a task, does the plan stay inside the issue's non-goals) is a mechanical cross-reference — so it's gated to the one class of task where this project's actual bug history says the stakes are highest.

## The Implementer (fresh job per task)

A **new Implementer job runs for each task** in the Planner's breakdown, via a GitHub Actions `strategy.matrix` populated from `needs.planner.outputs.tasks` (`max-parallel: 1` to preserve the sequential per-task dependency — a later task's job checks out the branch with every earlier task's commits already on it — `fail-fast: true` so a broken task stops the remaining ones rather than compounding). Each job:

1. Gets just its one task, plus the plan and analysis files for context (read from the checked-out branch, not passed inline).
2. Implements it using Superpowers' TDD skill: failing test, minimal code, passing test, commit — as separate commits, per the git-handoff section above.
3. Is reviewed **two ways before the next task's job runs**: spec compliance (does this task actually do what the plan said) and code quality (using Superpowers' code-review skill, or this environment's `/code-review` skill). A critical finding from either blocks moving to the next task.
4. Gets **one bounded self-fix attempt** against review findings, per the failure-handling rule above — not an open-ended back-and-forth with the reviewer.

This is a meaningfully more thorough process than one Implementer job working through an entire multi-task plan in a single session — each increment is verified before the next one is built on top of it, and a review finding is caught (and cheap to fix) at the size of one task rather than surfacing later as a tangle across the whole diff. It costs more invocations than a single-pass model would, which is a reasonable trade for scraping-touching work specifically; for small pure-logic plans with only one or two tasks, the practical difference is minor.

**`implementer.md` deliberately does not tell the job to read `docs/AGENT_HARNESS.md`.** A cost pass found this doc had grown to ~11K tokens — close to the entire measured cache-write cost of a fresh Implementer job's first turn — and most of it (resume logic, other stages' own sections, cost levers, observability) has no bearing on implementing one task. Every operational rule an Implementer actually needs (TDD flow, push/commit-title discipline, review expectations, the live-network and workflow-file restrictions) is inlined directly in `implementer.md` itself instead. This matters more here than for any other stage because the Implementer is the one dispatched repeatedly — up to once per task in the plan — so an unnecessary doc-read is a cost paid many times per run, not once. The other stage files still point at this doc because they're each run at most once per pipeline run, where the same waste doesn't compound.

## Deployer

Runs after the Tester, before the PR Risk Analyzer script. Composes the PR description — synthesized from `.agents/*.md` and the individual stage commit messages, not a dump of their raw content — then does the pre-squash `.agents/` cleanup described above. Rather than always defaulting to "open a PR," it picks from the same real choice Superpowers' branch-finishing step offers:

**The PR description includes a `Closes #N` line for the issue this run was dispatched for**, so merging the PR auto-closes the issue — no separate manual close step, no issue silently sitting open after its PR lands. This only applies when the run's diff actually closes out the issue's full scope; a partial pass (see issue #2's own history, split across two PRs by design) uses plain issue-referencing text instead of the auto-close keyword, since auto-closing an issue that still has real remaining scope would be actively misleading.

**The PR description does not include a risk-assessment section.** PR Risk Analyzer runs after the Deployer (it needs a real PR number to comment on), so there's nothing for the Deployer to read or synthesize at this point. The description says plainly that a risk determination follows as a separate PR comment once the PR exists, rather than guessing at one.

- **Open (or update) a PR** — the default outcome for most runs; the risk script's determination lands as a follow-up comment on whatever PR this step opens or updates.
- **Keep the branch, no PR yet** — for a run where the pipeline completed but the Deployer isn't confident enough to ask for review — the branch and its full `.agents/*.md` trail (pre-cleanup, in this case — no PR means no squash yet, so there's nothing to lose by leaving them) sit there for a human to look at directly. (No PR also means no PR Risk Analyzer run — it needs a PR number, so this path skips that stage entirely.)
- **Discard** — if something upstream made clear the whole task should be abandoned, rather than leaving a stale branch around indefinitely.
- **Merge is not an available choice right now** — see the non-goals below. This is a "for now," not a permanent stance: the project's author is generally open to real auto-merge once the PR Risk Analyzer's track record (from `.github/pr-risk-log.jsonl`) earns that trust.

**Opens/updates/keeps/discards, reports the PR number, and stops.** A human clicks merge. Until real auto-merge is trusted, the actual enforcement mechanism should be **GitHub branch protection requiring human review before merge** — not merely "the Deployer wasn't given a merge tool." Tool-list omission is a convention easily defeated by a Bash-equipped agent finding another way to call the GitHub API; branch protection is enforced by GitHub itself regardless of what any step tries.

Two jobs run this same logic depending on which path the run took — `deployer` (no live check needed) and `deployer-after-live-check` (after an approved live check) — both dispatching the same `deployer.md` file with the same steps, differing only in `needs:`/checkout ref and the prompt's framing of what just happened. See "Open questions," below, for the plan to de-duplicate this.

## PR Risk Analyzer

Runs after the Deployer, using the PR number it reported. **Determines and logs a risk level. Does not act on it — no auto-merge, no label-based merge trigger, nothing that touches merge state.** This is intentionally a dry run: the goal is to build a track record (how often would the low-risk calls have actually held up, with no follow-up fixes needed) before ever considering flipping on real auto-merge for real.

### Why this is a script, not a subagent

This stage used to be a Haiku subagent applying a "fixed rubric, not a subjective call." Looked at honestly, "fixed rubric evaluated against `git diff` output" isn't a lighter-weight judgment call, it's not a judgment call at all: every input (file list, line counts, hunk line-ranges, dependency/workflow-file touches, presence of a `test/` path in the diff) is something `git diff` already produces as plain, parseable text, and every rule below is a deterministic condition over that text. A model call here was paying for probabilistic text generation to do a job regular code does exactly and reproducibly, for free, every time. `scripts/pr-risk-check.js` is the deterministic replacement: same rubric, same two output artifacts, zero model cost, and — unlike a model call — the exact same diff always produces the exact same determination.

The determination is a fixed rubric, evaluated in code:

**HIGH RISK if any of:**
- Touches the load-bearing scraping/SMTP code called out in `CLAUDE.md` (checked via hunk-header line ranges in `check-wishlist.js`'s diff against the hardcoded load-bearing line bounds in the script — update those constants if the file's structure changes)
- Touches workflow YAML permissions, secrets, or the state-file commit/push step (any `.github/workflows/*` path in the diff)
- Adds or changes a dependency (`package.json` in the diff)
- Diff is large (>100 lines or >3 files)
- No test coverage for what changed (no `test/` path in the diff)

**LOW RISK requires all of:** pure-logic change only, small focused diff, adequate test coverage, all required checks green, no dependency/permission/secret changes — i.e. none of the above rules fire.

Two output artifacts, both reusing patterns already established in this repo rather than inventing new infrastructure:

1. A **PR comment** (via `gh pr comment`) stating the risk level and which rubric items fired — human-visible immediately, so a reviewer can sanity-check the call before merging.
2. An append to **`.github/pr-risk-log.jsonl`** (one line per PR: `{pr, sha, risk, reasons, checks_passed, timestamp}`) — same "commit a small file back to the repo" idiom as `notified.json`. This is the evidence base for eventually deciding auto-merge is trustworthy. The script commits and pushes this itself.

Invoked as a plain workflow step in the Deployer job, not a subagent: `node scripts/pr-risk-check.js <issue-number> <pr-number>`, using the PR number Deployer just reported (re-derived independently via `gh pr list --head <branch>`, not passed through as a job output).

## Cost levers beyond model choice

- **`effort` parameter** — same model, tunable reasoning depth, set via the `effort:` subagent frontmatter field. Wired in for the Analyzer specifically (`effort: low`) — it's on Haiku because its work is mechanical rubric-application rather than open-ended judgment; the model choice and the effort choice are two expressions of the same underlying reasoning, not independent levers. PR Risk Analyzer used to be the other stage in this bucket, but its rubric-application turned out to be fully deterministic rather than merely mechanical — see its own section above — so it's a plain script with no `effort` parameter to set at all now, the cheapest tier there is. Deliberately *not* set on Planner, Plan Validator, Implementer, Tester, or Deployer — those stages' whole value is thoroughness, and cutting effort there is exactly the wrong place to save money.
- **Prompt caching — real, but now scoped to within a single job, not across stages.** Each stage starts a genuinely fresh session, so there's no cross-stage cache reuse to rely on the way the old single-session design occasionally got (a second Implementer dispatch inside the same long-running session could hit a real cache hit on shared system-prompt content). That's a deliberate trade, not an oversight — see "How this actually runs," above: the old design's cross-stage caching was riding on the same growing-context mechanism that made its overall cost compound with how far into the run a stage happened to run. Within one job, caching still works exactly as normal for a multi-turn Claude Code session (the standard 5-minute ephemeral cache tier — confirmed from real production logs, not the 1-hour tier, which isn't exposed as a configurable option at the `claude_args` level anyway).

## Observability

The default `claude-code-action` invocation is nearly silent in the Actions log — "Running Claude Code via SDK (full output hidden for security)" until one final summary line with total cost and turn count. **`show_full_output: 'true'`** is set on every `claude-code-action` step across every job, so the raw turn-by-turn stream actually lands in the Actions log instead of being suppressed.

**Every real PR from this pipeline carries a "Run Cost" section**, appended after the Deployer job's own session ends — `scripts/compose-run-cost.js`, run as a plain workflow step, not a subagent. Because every stage is its own job, this is no longer an estimate: each single-instance stage's `total_cost_usd` is a plain GitHub Actions job output, and `compose-run-cost.js` composes the final PR table directly from those (`COST_ANALYZER`, `COST_PLANNER`, etc., passed as `"cost_usd,num_turns"` env-var pairs — plain env vars rather than a pre-built JSON blob, so the workflow YAML doesn't need inline JSON composition with its escaping pitfalls). The Implementer matrix is the one exception: GitHub Actions doesn't cleanly aggregate a matrix job's per-instance outputs (only the last instance's outputs survive to `needs.<job>.outputs`), so each Implementer task instance commits its own line to `.github/pipeline-run-log.jsonl` instead (`scripts/log-stage-cost.js`) — deliberately *not* under `.agents/`, since Deployer's pre-squash cleanup must not delete it before the final report reads it. `compose-run-cost.js` reads that file directly for the Implementer rows. Because the same `COST_*` env vars are threaded into both `deployer` and `deployer-after-live-check`, the cost report is complete regardless of which path a run took — unlike an earlier design this pipeline used briefly, where the live-check path's second job only had its own session's cost available and had to say so explicitly.

**A known fragility**: the execution-output file path (`$RUNNER_TEMP/claude-execution-output.json`) is an internal `claude-code-action` implementation detail observed in real run logs, not a documented, versioned contract. The cost-extraction scripts degrade gracefully (report zero/empty rather than failing the job) if it's not where expected, but this could silently stop working on a `claude-code-action` version bump — worth spot-checking after any update to that action's pinned version.

**A second fragility in the same file**: its top-level JSON shape isn't fixed either — sometimes it's the single final `"result"` object directly (`{total_cost_usd, num_turns, ...}`), sometimes (with `show_full_output: true`) a JSON array of every stream event from the session, with the `result` event as one entry rather than the whole file. Every jq-based "Extract cost" step now runs through a small filter that unwraps either shape (`if type=="array" then (find .type=="result", else last element) else . end`) before reading fields, and `log-stage-cost.js` has the equivalent `extractResult()` helper on the Node side. This was found the hard way — see the debugging-history section below — and silently produced empty cost/turn outputs before being caught, because a failing `jq` inside `echo "x=$(jq ...)"` doesn't trip `bash -e`.

## Explicit non-goals

- No price-drop override or other scope creep on the dedup feature (see issue #1's own non-goals — this harness doesn't relitigate them).
- No database, no external service, no GitHub Actions cache/artifacts for pipeline state — git commits only, per "Git as the inter-stage handoff mechanism" above.
- No auto-merge until the PR Risk Analyzer has an actual track record to point to.
- No stage spawns another stage directly — GitHub Actions' own job graph (`needs:`) is the only dispatch mechanism; there's no recursive delegation to guard against because there's no in-process dispatch at all.
- No trigger mechanism beyond the Actions-tab `workflow_dispatch` form — no issue labels, no `issue_comment` webhook workflow. One button, one field, every time.

## Debugging history

Real bugs found across real dogfooded runs, kept here because the reasoning behind each fix is worth more than the diff alone. Roughly chronological.

**The Orchestrator's own turn could be cut off mid-task** (single-session design) — specifically, mid-way through a Tester dispatch that had already run real tests (13/13 passing), by whatever mechanism forces a `--json-schema` structured-output answer, before it could push anything or post a comment. `--max-turns` turned out not to be the cause (Claude Code has no default turn limit; the docs for structured outputs separately mention a model-fallback event can retract in-progress output mid-stream, which fits what was observed). Two fixes, both carried forward into every job in the current design: **`--max-turns 150`** as a real cost/runaway safety net regardless of cause, and **`--disallowedTools "ScheduleWakeup,mcp__*"`** — the same run's log showed an attempt to call `ScheduleWakeup`, a tool from an entirely different product (an interactive session's own wakeup-scheduling mechanism) that makes no sense in a one-shot ephemeral Actions job. It's exposed via `claude-code-action`'s own `enableAllProjectMcpServers: true` default, which pulls in whatever MCP servers happen to be configured in the runner's environment — this pipeline needs none of them (Superpowers is skills-only, no MCP server), so blocking `mcp__*` entirely, plus the specific non-namespaced `ScheduleWakeup`, removes an entire class of tools that shouldn't be in a stage's context in the first place.

**Stale-branch drift**, first real run against issue #17 under the split architecture: every job past Analyzer checks out `needs.analyzer.outputs.branch` directly and runs `scripts/*.js` from that checkout — but the issue-17 branch had been created (during earlier 2-job-slice prototyping) *before* the full pipeline's own scripts (`scripts/log-stage-cost.js`, `scripts/compose-run-cost.js`) were added to `main`. The Implementer job's real work completed fine; the very next step, `node scripts/log-stage-cost.js ...`, failed with `MODULE_NOT_FOUND` because that script simply didn't exist in that branch's tree yet. Fix: Analyzer now merges `origin/main` into a *resumed* branch and pushes, before any other job checks it out — see "Fresh start vs. resume," above.

**`claude-code-action` revokes its own installation token right after the session ends**, breaking any *later* step's own `git push`. It mints a scoped GitHub App installation token for the Claude session's own git operations, then revokes it (a `curl -X DELETE .../installation/token` visible in the job log) as an inline part of its own composite-action cleanup, immediately after the session ends and before control returns to the next declared workflow step. This broke `scripts/log-stage-cost.js`'s post-session push first (`Invalid username or token. Password authentication is not supported for Git operations.`), then `scripts/pr-risk-check.js`'s identically, once that was fixed — both do their own `git push` as a plain workflow step *after* the `claude-code-action` step, unlike everything that pushes *during* a session (which works fine) or *before* one (the "Sync resumed branch" step, which also works fine). Fix: both scripts gained a `reauthOrigin()` helper that re-points `origin` at an explicit `GH_TOKEN`-authed URL immediately before their own push, sidestepping whatever credential state `claude-code-action` left behind.

**GitHub Actions' implicit `success()` considers the whole transitive dependency graph, not just a job's own direct `needs:`.** `tester`'s condition was originally a bare expression (`if: needs.implementer.result == 'success'`), and GitHub Actions prepends an implicit `success()` to any job `if:` not already wrapped in `always()`/`success()`/`failure()`. When `plan-validator` correctly sat out a run (not scraping-touching), that skip cascaded through `tester`, `deployer`, `live-amazon-check`, and `deployer-after-live-check` even though Analyzer, Planner, and every Implementer task had genuinely succeeded — exactly the class of problem `implementer`'s own `if: always() && ...` was already written to guard against, just not yet applied to the other four conditional jobs. Fixed by adding `always() &&` to all four.

**A legitimately empty task list looks identical to a failed Implementer, twice over.** Once every task in a plan has already been completed by earlier runs, the Planner — noticing this itself from git history and the existing `.agents/plan.md` — correctly reports `tasks: []` rather than re-emitting a stale breakdown against preconditions that no longer hold. Two separate consequences followed: first, `tester`'s condition only accounted for `needs.implementer.result == 'success'`, so a legitimately empty task list skipped everything downstream just like a real Implementer failure would have (fixed: `tester` also treats `needs.planner.outputs.tasks == '[]'` as a valid reason to proceed). Second, and less obviously, a GitHub Actions matrix job with zero entries produces *zero job instances at all* — not even a skipped one — and this specifically makes GitHub mark the entire workflow run's own top-level status as "failure," even though nothing downstream actually broke (confirmed by comparing two separate runs that both hit this case, both showing run-level "failure" despite identical all-success job conclusions on everything that ran). Fixed: `implementer` now has an explicit `if:` guard (`needs.planner.outputs.tasks != '[]'`) so an empty task list skips the whole job the ordinary way, like `plan-validator` already does, instead of asking GitHub Actions to expand a zero-entry matrix at all.

**The execution-output file's JSON shape isn't fixed** — see "Observability," above, for the fix. Found because every jq-based cost extraction across the pipeline was silently producing empty cost/turn outputs on a real run, without ever failing the job (a failing `jq` inside `echo "x=$(jq ...)"` doesn't trip `bash -e`).

## Open questions (tracked in follow-up issues, not resolved here)

- **No skip-if-already-done resume logic.** See "Fresh start vs. resume," above. Solving this properly means each job checking whether its own artifact already exists on the branch and, if so, reading committed state instead of re-dispatching — a real, separate-sized piece of work, not a quick patch.
- **Duplicated Deployer job definition.** `deployer` and `deployer-after-live-check` are near-identical (same steps, different `needs:`/checkout ref and prompt framing). GitHub Actions' reusable-workflow feature (`workflow_call`) is the right tool to de-duplicate this, not yet used anywhere in this repo.
- The `live-amazon-check` job depends on a GitHub Environment named `live-amazon-check` with required reviewers actually being configured in repo Settings — a manual step this repo's own automation can't do on its own; verify it's set before trusting that gate, don't assume it's done.
